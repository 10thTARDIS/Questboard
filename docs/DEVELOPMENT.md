## First-time Setup

```bash
# 1. Clone the repository
git clone https://github.com/10thTARDIS/Questboard quest-board
cd quest-board

# 2. Copy and edit the environment file
cp .env.example .env
# Edit .env — at minimum set OIDC_* and change database passwords

# 3. Install pre-commit hooks
pre-commit install

# 4. Start all services
docker compose up --build

# 5. Run database migrations (first time only)
docker compose exec backend alembic upgrade head
```

The app is now available at:
- **Frontend (Vite dev server):** http://localhost:5173
- **Backend API:** http://localhost:8000
- **Interactive API docs:** http://localhost:8000/docs

## Running in Development

```bash
# Start all services (detached)
docker compose up -d

# View logs for a specific service
docker compose logs -f backend
docker compose logs -f worker

# Stop all services
docker compose down

# Rebuild after changing requirements.txt or package.json
docker compose up --build
```

### Hot-reload

- **Backend** — Uvicorn runs with `--reload`, so Python file changes are picked up automatically.
- **Frontend** — Vite's HMR updates the browser instantly on `.jsx`/`.css` changes.
- **Celery worker** — Does **not** hot-reload. Restart after task code changes:
  ```bash
  docker compose restart worker beat
  ```

## Environment Variables

Copy `.env.example` to `.env` and fill in the values. The complete reference is in `README.md`, but the most important settings are:

| Variable | Description |
|----------|-------------|
| `APP_URL` | Public URL of the frontend (e.g. `http://localhost:5173` in dev) |
| `SECRET_KEY` | Random secret — generate with `openssl rand -hex 32` |
| `INVITE_CODE` | Optional registration gate. Leave blank for open registration |
| `DATABASE_URL` | asyncpg connection string for FastAPI |
| `DATABASE_MIGRATE_URL` | psycopg2 connection string for Alembic |
| `OIDC_DISCOVERY_URL` | URL of the OIDC provider's `.well-known/openid-configuration` |
| `OIDC_CLIENT_ID` | OAuth 2.0 client ID |
| `OIDC_CLIENT_SECRET` | OAuth 2.0 client secret |
| `OIDC_REDIRECT_URI` | Must match the callback URL registered with the provider: `{APP_URL}/auth/callback` |
| `DEFAULT_DISCORD_WEBHOOK_URL` | Optional fallback webhook for campaigns without their own |

## Database Migrations

Migrations use [Alembic](https://alembic.sqlalchemy.org/).

```bash
# Apply all pending migrations
docker compose exec backend alembic upgrade head

# Roll back the last migration
docker compose exec backend alembic downgrade -1

# Check current revision
docker compose exec backend alembic current

# Generate a new migration after changing a model
docker compose exec backend alembic revision --autogenerate -m "add xyz column"
```

> **Note:** Always review autogenerated migrations before applying them. Alembic cannot detect all changes (e.g. column type changes may need manual adjustment).

### Migration conventions

- All ENUM types are created with raw `op.execute("CREATE TYPE ...")` before the tables that use them.
- The `server_default` for PostgreSQL ENUM columns must use `sa.text("'value'")` (with quotes inside the string), not a plain string.
- Drop in the reverse order of creation in `downgrade()`.

## Backend Conventions

### Layer separation

The backend follows a strict three-layer architecture:

```
Router (HTTP) → Service (business logic) → Database (SQLAlchemy)
```

- **Routers** (`app/routers/`) handle HTTP specifics: request parsing, response codes, `HTTPException`. They call services and translate `ValueError` → `HTTPException(400)`.
- **Services** (`app/services/`) contain all business logic. They take `AsyncSession` and domain objects; they raise `ValueError` for invalid operations. No `HTTPException` here.
- **Models** (`app/models/`) are pure SQLAlchemy ORM definitions.
- **Schemas** (`app/schemas/`) are Pydantic models for API input validation and output serialisation.

### Database sessions

The `get_db` dependency (in `database.py`) provides a session per request, commits on success, and rolls back on exception. Services should call `await db.flush()` to get auto-generated IDs before creating related rows, then `await db.commit()` at the end of a logical operation.

### Adding a new model

1. Create `app/models/your_model.py` with the SQLAlchemy class.
2. Import it in `alembic/env.py` (or ensure it is imported transitively via `app.database.Base`).
3. Generate a migration: `alembic revision --autogenerate -m "add your_model table"`.
4. Review and apply: `alembic upgrade head`.

### Adding a new endpoint

1. Add the route to the relevant router in `app/routers/`.
2. Create or extend the schema in `app/schemas/`.
3. Implement business logic in `app/services/`.
4. Register the router in `app/main.py` if it's a new file.

## Frontend Conventions

### State management

There is no global state library. State is managed with React's `useState` and `useEffect`:

- Authentication state is provided by `AuthProvider` / `useAuth` (context).
- Page-local state (campaign data, form fields, errors) is co-located in the page component.
- Optimistic updates are used where appropriate: new items are prepended to lists immediately without waiting for a re-fetch.

### API client

Each resource has a dedicated file in `src/api/`:

- `auth.js` — `/api/me`, `PATCH /api/me`
- `campaigns.js` — campaign CRUD, members, invite codes, milestones, notes
- `sessions.js` — sessions, time slots, votes, attendance, calendar
- `users.js` — platform links (`/api/me/platform-links`)

All files use a private `request()` helper that:
- Sends credentials (the cookie) with every request
- Throws an `Error` with the backend's `detail` message on non-OK responses
- Returns `null` for `204 No Content`

### Adding a new API call

Add a named export to the relevant `src/api/*.js` file. Follow the existing pattern of thin one-liner exports that call `request()`.

### Styling

Tailwind CSS (v3) with JIT mode. All classes are applied directly in JSX — no separate CSS files except `index.css` which imports Tailwind's base layers.

### Route structure

```
/login               → Login.jsx          (public)
/dashboard           → Dashboard.jsx      (protected)
/profile             → Profile.jsx        (protected)
/admin               → Admin.jsx          (protected, admin only)
/campaigns/:id       → CampaignDetail.jsx (protected)
/campaigns/:id/notes → CampaignNotes.jsx  (protected)
/sessions/:id        → SessionDetail.jsx  (protected)
*                    → redirect to /dashboard
```

All protected routes are wrapped in `<AuthGuard>` which redirects unauthenticated users to `/login`.

Pre-commit hooks run automatically on `git commit`. To run them manually:

## Adding a New Feature — End-to-End Walkthrough

This example adds a `color` field to campaigns.

### 1. Database migration

Add the column to the model (`app/models/campaign.py`):
```python
color: Mapped[str | None] = mapped_column(Text, nullable=True)
```

Generate and apply the migration:
```bash
docker compose exec backend alembic revision --autogenerate -m "add color to campaigns"
docker compose exec backend alembic upgrade head
```

### 2. Schema

Add `color` to `CampaignCreate`, `CampaignUpdate`, and `CampaignResponse` in `app/schemas/campaign.py`.

### 3. Service

`update_campaign` already uses `data.model_dump(exclude_unset=True)` so no changes needed there. `create_campaign` needs `color=data.color` added to the `Campaign(...)` constructor.

### 4. Router

No changes needed if the schema changes are sufficient.

### 5. Frontend

- Add a color input to the create/edit forms in `CampaignDetail.jsx` and `Dashboard.jsx`.
- Add `color` to the `createForm` initial state.
- Pass `color: createForm.color || null` in the API call body.

## Troubleshooting

### "Session expired or invalid" immediately after login

Check that `APP_URL` in `.env` matches the origin the browser is connecting from. If they don't match, the CORS middleware rejects the callback redirect and the session cookie is not set.

### Celery tasks not executing

1. Check the worker is running: `docker compose ps`
2. Check for errors in the worker logs: `docker compose logs worker`
3. Verify `CELERY_BROKER_URL` points to the correct Redis DB.

### Alembic "Target database is not up to date"

Run `alembic upgrade head`. If the migration history is corrupt, check `alembic current` and `alembic history` to diagnose.

### OIDC "Invalid or expired state parameter"

The PKCE state token has a 10-minute TTL. If the user takes longer than 10 minutes on the provider's login page, the callback will fail. This is expected behaviour. The user should click "Sign in" again.

### Discord webhook not sending

1. Verify the webhook URL starts with `https://discord.com/api/webhooks/`.
2. Check the Celery worker logs for HTTP errors.
3. Confirm the worker is connected to the correct Redis broker.
4. Ensure the campaign has a webhook configured, or `DEFAULT_DISCORD_WEBHOOK_URL` is set.
