# Quest Board — Project Build Plan

## Project Summary

Quest Board is a self-hosted web application for scheduling tabletop RPG sessions. It replaces
RollRota for a small, trusted group. The GM and players log in via OIDC (Authentik), propose and
vote on session times, and receive reminders via Discord webhook. The entire stack runs in Docker
Compose on a home server. TLS termination is handled externally by a cloud-based Caddy reverse
proxy — do not add any TLS/cert logic to this project.

```
Users
  id              UUID PK
  oidc_sub        TEXT UNIQUE NOT NULL       -- provider subject claim
  oidc_issuer     TEXT NOT NULL              -- issuer URL (supports multiple providers)
  display_name    TEXT NOT NULL
  avatar_url      TEXT
  email           TEXT
  created_at      TIMESTAMPTZ

CampaignMembers (join table)
  campaign_id     UUID FK → Campaigns
  user_id         UUID FK → Users
  role            ENUM('gm', 'player')

Campaigns
  id              UUID PK
  name            TEXT NOT NULL

  description     TEXT
  discord_webhook_url  TEXT                  -- nullable
  invite_code     TEXT UNIQUE                -- share to let players join
  created_at      TIMESTAMPTZ

Sessions
  id              UUID PK
  campaign_id     UUID FK → Campaigns
  title           TEXT
  description     TEXT
  scheduling_mode ENUM('vote', 'direct', 'tentative')
  status          ENUM('proposed', 'confirmed', 'completed', 'cancelled')
  confirmed_time  TIMESTAMPTZ                -- null until confirmed

  created_by      UUID FK → Users
  created_at      TIMESTAMPTZ

TimeSlots
  id              UUID PK
  session_id      UUID FK → Sessions
  proposed_time   TIMESTAMPTZ NOT NULL
  created_at      TIMESTAMPTZ

Votes
  id              UUID PK
  time_slot_id    UUID FK → TimeSlots
  user_id         UUID FK → Users
  availability    ENUM('yes', 'maybe', 'no')
  updated_at      TIMESTAMPTZ
  UNIQUE (time_slot_id, user_id)             -- one vote per slot per person
```

## Authentication Architecture

### OIDC Flow (Generic — works with Authentik, Keycloak, Google, etc.)

```
1. GET /auth/login
   → Backend generates state token + PKCE code_verifier
   → Stores (state → code_verifier) in Redis with 10-minute TTL
   → Redirects browser to provider's authorization_endpoint
     with response_type=code, scope=openid profile email,
     code_challenge (S256), state

2. Provider redirects to GET /auth/callback?code=...&state=...
   → Backend validates state exists in Redis (CSRF protection), deletes it
   → Exchanges code + code_verifier for tokens (PKCE prevents interception)
   → Fetches userinfo from provider's userinfo_endpoint
   → Creates or updates User record (keyed on iss + sub)
   → Creates server-side session in Redis with 8-hour TTL
   → Sets HttpOnly, SameSite=Lax, Secure session cookie
   → Redirects to frontend dashboard

3. Subsequent requests
   → Session cookie read by backend dependency get_current_user
   → Redis lookup returns user_id → DB lookup returns User
   → 401 if session missing or expired

4. GET /auth/logout
   → Deletes Redis session
   → Clears cookie
   → Redirects to login page
```

### Role Enforcement

- Roles are per-campaign (GM or player), not global
- `require_gm` dependency checks CampaignMembers for the current user
- All role checks happen server-side in FastAPI dependencies — never trust client claims
- GMs can: create/edit/cancel sessions, propose time slots, confirm sessions, post notes
- Players can: vote on time slots, view sessions they're members of

## Session Scheduling Modes

All three modes use the same underlying data model. Mode determines the creation flow:

| Mode        | Description                                                         | Voting? |
|-------------|---------------------------------------------------------------------|---------|
| `vote`      | GM proposes 2–5 slots; players vote; GM confirms a winner           | Yes     |
| `direct`    | GM picks a single time immediately; session confirmed on creation   | No      |
| `tentative` | GM sets a likely time but flags it unconfirmed; can upgrade later   | No      |

For `direct` and `tentative`, a single TimeSlot is created automatically. For `direct`,
the session `status` is set to `confirmed` and `confirmed_time` is populated immediately.
Discord notification fires at creation time for `direct` mode.

---

## Voting Logic

- Players mark each slot: `yes` / `maybe` / `no`
- Score per slot: `yes` × 2 + `maybe` × 1
- Best slot = highest score (ties broken by fewest `no` votes)
- Frontend highlights the best slot; GM makes the final call
- Once GM confirms a slot: `sessions.status` → `confirmed`, `sessions.confirmed_time` set,
  all scheduled reminder tasks created, Discord confirmation notification sent
- Votes are mutable until the session is confirmed

## Notification Architecture

### Phase 1: Discord Webhooks

Notifications are sent by Celery tasks. A `NotificationBackend` Protocol is defined now
so additional platforms (email, Matrix) can be added in Phase 2 without refactoring.

```python
class NotificationBackend(Protocol):
    async def send_session_proposed(self, session: Session, campaign: Campaign) -> None: ...
    async def send_session_confirmed(self, session: Session, campaign: Campaign) -> None: ...
    async def send_session_reminder(self, session: Session, campaign: Campaign, hours_until: int) -> None: ...
    async def send_session_cancelled(self, session: Session, campaign: Campaign) -> None: ...
```

Discord messages should use rich embeds, not plain text.

### Reminder Schedule (Celery beat)

When a session is confirmed, schedule three Celery tasks with ETAs:
- 7 days before `confirmed_time`
- 24 hours before `confirmed_time`
- 1 hour before `confirmed_time`

Store Celery task IDs on the Session record so they can be revoked if the session is
rescheduled or cancelled.

## Security Requirements (Non-Negotiable)

### Secrets & Configuration
- All secrets in environment variables only — never in code or committed files
- `.env` in `.gitignore`; `.env.example` documents every variable with safe placeholder values
- `SECRET_KEY` must be generated with `openssl rand -hex 32`

### Authentication
- OIDC with PKCE — no passwords ever stored in this application
- State parameter validated and deleted from Redis before use (prevents replay)
- Sessions stored server-side in Redis; cookie contains only an opaque session ID
- Cookies: `HttpOnly=True`, `SameSite=Lax`, `Secure=True`
- Session TTL: 8 hours; refresh on activity

### Transport & Headers
- Assume HTTPS is enforced upstream — do not handle TLS here
- FastAPI middleware must set: `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`,
  `Referrer-Policy: strict-origin-when-cross-origin`, `Content-Security-Policy` (strict)

### API Security
- Rate limiting on all `/auth/*` endpoints via `slowapi` (10 req/min per IP)
- All mutations require authentication; all campaign-scoped endpoints require membership
- CORS: allow only `APP_URL` origin, credentials=True
- Parameterised queries only — SQLAlchemy ORM handles this; raw SQL is forbidden

### Registration Gating
- `INVITE_CODE` env var gates account creation
- If set, the OIDC callback checks the code before creating a new user record
- Existing users are always allowed in regardless of invite code

### Infrastructure
- All Docker containers run as non-root users
- Database user (`rollcall`) has SELECT, INSERT, UPDATE, DELETE only — no DDL privileges
- Alembic migrations run as a separate privileged user (`rollcall_migrate`)
- No secrets in Docker image layers — use runtime env vars only

### Dependencies
- Pin all dependencies (use `pip-compile` for backend, exact versions in `package.json`)
- Add `pip audit` as a pre-commit hook and document it in README

## Version 2 Features (Design Now, Build Later)

### Reaction-Based Voting via Bot
- Users link their Discord (or Matrix) account ID in profile settings
  (stored as `platform_links` table: user_id, platform ENUM, platform_user_id)
- When a vote notification is posted, bot adds one reaction emoji per time slot
- Bot watches for reactions from users with linked accounts and writes votes
  back to the existing Votes API
- Users without linked accounts see a prompt when they react

### Session Recording & Transcription
- Discord bot joins voice channel on session start (opt-in, consent required)
- Audio sent to self-hosted OpenAI Whisper for transcription
- Transcript passed to LLM (Ollama or API) for session summary
- Transcript and summary stored on Session record, displayed in session detail page

## Key Technical Decisions & Rationale

| Decision | Choice | Reason |
|---|---|---|
| Auth | Generic OIDC, no local passwords | Authentik handles all credential management; no password storage risk |
| Sessions | Server-side in Redis | Instant revocation; no sensitive data in cookie |
| PKCE | Required | Prevents auth code interception even if redirect URI is compromised |
| Async SQLAlchemy | `asyncpg` driver | Consistent async throughout; better FastAPI integration |
| Celery task IDs stored | On Session model | Enables clean revocation when sessions change |
| NotificationBackend Protocol | Defined in Phase 1 | Prevents refactoring when adding email/Matrix in Phase 2 |
| UUIDs as PKs | All tables | Avoids enumeration attacks on IDs in URLs |
| Two DB users | app + migrate | App user cannot modify schema; limits blast radius of SQL injection |
